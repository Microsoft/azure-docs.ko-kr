## <a name="what-is-managed"></a>' 관리 '

기본 IaaS 클라우드 구성 요소와 달리 계산, 네트워킹 등 하위 수준 컨트롤 및 사용자 지정 옵션을 활용 하 여 사용자를 노출 하는 AKS 서비스 구성의 공통 집합을 나타내는 턴키 Kubernetes 배포를 제공 하 고 Kubernetes에 필요한 기능입니다. 이 서비스를 사용 하는 고객 사용자 지정, 배포 및 기타 옵션의 제한 수입니다. 즉, 고객은 걱정할 또는 Kubernetes 클러스터를 직접 관리할 필요가 없습니다.

고객 AKS를 통해 완전히 관리 되는 가져옵니다 **제어 평면** – 제어 평면 모든 구성 요소와 작동 하 고 최종 사용자에 게 Kubernetes 클러스터를 제공 하는 데 필요한 서비스를 포함 하는 위치입니다. Kubernetes에 대 한 구성 요소를 모두 유지 관리 되며 Microsoft에서 운영 됩니다.

관리 되는 **제어 평면** 다음 구성 요소는 관리 하 고 Microsoft에서 모니터링 합니다.

* Kubelet / Kubernetes API 서버
* Etcd 또는 서비스, 확장성 및 런타임의 품질을 포함 하 여 – 호환 키/값 저장소
* DNS 서비스 (예를 들어, kube-dns / CoreDNS)
* Kubernetes 프록시/네트워킹

AKS 관리 되는 100% 아닙니다 **클러스터** 솔루션입니다. 특정 구성 요소 (예: 작업자 노드)는 특정 **공동 책임** 경우 AKS 클러스터를 유지 관리 작업에서 사용자 상호 작용을 필요로 하는 위치입니다. 작업자 노드 예를 들어, OS 보안 패치 응용 프로그램

 **관리 되는**, 즉 Microsoft와 AKS 팀 배포의 작동 및 가용성과 이러한 서비스의 기능을 담당 합니다. **고객은 이러한 구성 요소를 변경할 수 없습니다**합니다. 사용자 지정 일관적이 고 확장성이 뛰어난 사용자 환경을 보장 하려면 제한 됩니다. 완전히 사용자 지정 가능한 솔루션을 참조 하세요 [AKS 엔진](https://github.com/Azure/aks-engine)합니다.

> [!NOTE]
> 사용자 지정 Azure 리소스 그룹에 이러한 가상 컴퓨터를 배포 하지만 일반 Azure IaaS 리소스와 Azure Kubernetes Service 작업자 노드 Azure Portal에 표시 되는지 이해 해야 (MC 접두사로\\*). 그러나 사용자 변경 될 수 있습니다, SSH에 기본 가상 컴퓨터와 마찬가지로 (기본 OS 이미지를 변경할 수 없는 및 변경 내용 업데이트를 통해 유지 되지 수도 다시 부팅)에 다른 Azure 리소스를 연결 하거나 그렇지 않은 경우 수정할 수 있습니다. **그러나 이렇게의 대역 외 관리 및 사용자 지정 자체 AKS 클러스터를 지원 될 수 있습니다 의미 합니다. Microsoft 지원의 지시가 없는 한 작업자 노드 변경 하지 마세요.**

## <a name="what-is-shared-responsibility"></a>새로운 공동 책임

AKS는 클러스터 생성 시 고객에 의해 정의 된 Kubernetes 작업자 노드의 수를 만듭니다. 이러한 노드를 설명한 것 처럼 고객 워크 로드 실행 되는 경우 고객 소유 하 고 보거나 수정할 수 있습니다 이러한 작업자 노드.

Microsoft 지원에 이러한 노드 전용 코드를 실행 하는 중요 한 데이터를 저장 하기 때문에 **제한 된 액세스** 모든 고객에 게 클러스터 노드. Microsoft 기술 지원 서비스에 로그인 하 고 명령을 실행 하 하거나 express 고객 사용 권한 및/또는 지원을 지원 팀에서 지시한 대로 디버깅 명령을 실행 하지 않고 이러한 노드에 대 한 로그를 볼 수 없습니다.

이러한 작업자 노드의 중요 한 특성상 Microsoft는 이러한 노드의 '백그라운드' 관리 제한에 주의 사용 합니다. Kubernetes 마스터 노드와 etcd, (Microsoft에서 관리 되는) 다른 구성 요소 실패 하는 경우에 대부분의 경우에서 실행 작업 계속 합니다. 작업자 노드는 주의 하지 않고 수정 되 면 데이터 및/또는 작업 손실를 클러스터 못했고 렌더링 합니다.

## <a name="azure-kubernetes-service-support-coverage"></a>Azure Kubernetes Service 지원 적용 범위

**Microsoft는 다음에 대 한 기술 지원을 제공 합니다.**

* 제공 하 고 Kubernetes 서비스 (예: API 서버)에서 지 원하는 모든 Kubernetes 구성 요소에 연결
* 관리, 가동 시간, QoS 및 Kubernetes 작업 제어 평면 서비스 (Kubernetes 마스터 노드, API 서버 etcd, kube-예를 들어 dns입니다.
* Etcd 지원 모든 etcd 데이터에 대해 투명 하 게 자동화 된 백업을 30 분 마다 재해 계획 및 클러스터 상태 복원이 포함 되어 있습니다. 이러한 백업은 고객/사용자에 게 직접 제공 하지 않으며 데이터 안정성과 일관성을 확인 하는 데 사용 됩니다.
* 영구적 볼륨, 네트워킹 (Kubernetes 및 Azure CNI) 부하 분산 장치 같은 서비스를 다른 Azure 통합 같은 Kubernetes에 대 한 Azure Cloud Provider 드라이버에서 모든 통합 지점
* Kubernetes API 서버, 등 etcd, kube-dns 제어 평면 구성 요소의 사용자 지정 관련 문제 또는 질문 합니다.
* Azure CNI, Kubenet, 또는 다른 네트워크 같은 네트워킹 항목에 대 한 액세스 및 기능 문제 (DNS 확인, 패킷 손실, 라우팅 및 등)을 발급 합니다.
  * 클러스터 및 관련된 구성 요소, 다른 Azure 서비스 및 응용 프로그램에 대 한 연결 내에서 Kubenet (기본) 및 (Azure CNI) 네트워킹 고급를 포함 하는 네트워킹 시나리오를 지원 합니다. 또한 Microsoft에서 수신 컨트롤러 및 수신/부하 분산 장치 구성, 네트워크 성능 및 대기 시간 지원 됩니다.

**Microsoft는 다음에 대 한 기술 지원을 제공 하지 않습니다.**

* 권고 / "방법" 사용 하 여 Kubernetes 질문, 예를 들어 응용 프로그램 워크 로드 질문에 대 한 사용자 지정 수신 컨트롤러를 만드는 방법 및 세 번째-파티/OSS 패키지 또는 도구에서 다루지 않습니다.
  * AKS에 대 한 자문 티켓 클러스터 기능을 사용자 지정 튜닝-예: Kubernetes 작업 문제/방법 tos 범위 내에 있습니다.
* Kubernetes의 일부로 제공 되지 않습니다 타사 오픈 소스 프로젝트 제어 평면 또는 Istio, Helm, Envoy 등과 같은 AKS 클러스터를 사용 하 여 배포 합니다.
  * 예제 및 Microsoft 설명서에서 및 Kubernetes Azure 클라우드 공급자와 해당 타사 오픈 소스 도구 통합 하는 위치를 제공 하는 응용 프로그램에 대 한 최상의 지원이 제공 됩니다 Kured, Helm 등의 타사 오픈 소스 프로젝트에 대 한 또는 다른 AKS 관련 버그입니다.
* 타사 닫힌 소스 소프트웨어 도구, 네트워킹 장치 또는 소프트웨어를 검색 하는 보안을 포함할 수 있습니다.
* "다중 클라우드" 또는 다중 공급 업체 빌드 제한에 대 한 문제는 지원 되지 않습니다, 예를 들어 페더레이션 다중 공용 클라우드 공급 업체 솔루션을 실행 중인 지원 되지 않습니다.
* 공식에서 설명 하는 것 이외의 특정 네트워크 사용자 지정 [AKS 설명서][3]합니다.
  > [!NOTE]
  > 문제 및 네트워크 보안 그룹 관련 버그 지원 됩니다. 예를 들어 지원 Nsg를 업데이트 하려면 실패 또는 예기치 않은 NSG 또는 부하 분산 장치 동작 관련 질문에 대답할 수 있습니다.

## <a name="azure-kubernetes-service-support-coverage-worker-nodes"></a>Azure Kubernetes 서비스 지원 지역 (작업자 노드)

### <a name="microsoft-responsibilities-for-azure-kubernetes-service-worker-nodes"></a>Azure Kubernetes Service 작업자 노드에 대 한 Microsoft의 책임

설명한 대로 `shared responsibility` 섹션, 작업자 노드 공동 책임 모델에 해당 하는 Kubernetes 위치:

* 필요한 추가 기능 (예: 모니터링 및 에이전트 네트워킹)를 사용 하 여 기본 운영 체제 이미지를 제공 합니다.
* 운영 체제 패치 작업자 노드로 자동 배포
* Kubernetes 사용 하 여 문제를 자동으로 재구성 제어 평면 구성 요소와 같은 작업자 노드에서 실행 합니다.
  * kube-proxy
  * Kubernetes를 통신 경로 제공 하는 네트워킹 터널 마스터 구성 요소
  * kubelet
  * docker/모 비 데몬

> [!NOTE]
> 제어 평면 구성 요소는 작업자 노드에서 작동 하지 않는 경우 AKS 팀 문제를 해결 하려면 전체 작업자 노드를 다시 부팅 해야 합니다. 사용자 대신 수행 이며 고객 에스컬레이션 (고객에 게 활성 작업 및 데이터에 대 한 액세스로 인해) 항목이 있어야만 수행 되지 않습니다. 가능한 AKS 담당자는 때마다 비 응용 프로그램에 영향을 주는 있도록 필요한 작업을 다시 부팅 합니다.

### <a name="security-issues-and-patching"></a>보안 문제 및 패치

경우에 따라 (예: CVEs) 보안 결함 AKS 서비스의 구성 요소 중 하나 이상 찾을 수 있습니다. 이러한 시나리오에서는 AKS는 가능 하면이 문제를 완화 하려면 영향을 받는 모든 클러스터를 패치 또는 사용자에 게 업그레이드 지침을 제공 합니다.

이러한 보안 허점을, 영향을 받는 작업자 노드에 대 한 AKS 팀 핫픽스를 적용 되며 변경의 사용자에 게 알림 문제의 가동 패치를 사용할 수 있는 경우.

보안 패치를 요구 하는 경우 작업자 노드 다시 부팅, Microsoft는이 요구 사항의 고객에 게 알립니다. 및 해당 클러스터에 대 한 패치를 가져오려는 업데이트 또는 다시 부팅을 실행 하는 고객 담당 합니다. 사용자 AKS 지침에 따라 패치 적용 되지 않은 경우에 문제에 취약 하 게 해당 클러스터는 계속 됩니다.

### <a name="node-maintenance-and-access"></a>노드 유지 관리 및 액세스

작업자 노드는 책임을 공유 하기 때문에 및 고객의 소유권을 고객에 게 이러한 작업자에 로그인 하 고 패키지를 제거 하 고 새 패키지를 설치, 커널 업데이트와 같은 잠재적으로 해로운 변경 작업을 수행 수 있습니다.

고객은 안전 하지 않은 작업 또는 오프 라인으로 전환 하려면 제어 평면 구성 요소를 트리거하는 작업을 수행 하거나 작동 하지 않게 그렇지 않은 경우, AKS 서비스는이 오류를 감지 하 고 이전 작업에 작업자 노드를 복원 하려면 자동 업데이트 관리 수행 상태입니다.

고객에 게 로그인 하 고 alter 작업자 노드 수, 이지만 *권장 되지 않습니다* 클러스터를 못했고 내릴 수 있습니다이 있기 때문입니다.

## <a name="network-ports-access-and-network-security-groups"></a>네트워크 포트, 액세스 및 네트워크 보안 그룹

관리 되는 서비스로 AKS는 특정 네트워킹 및 연결 요구 사항이 있습니다. 이러한 요구 사항은 일반 IaaS 구성 요소 보다 덜 효율적입니다. 다른 IaaS 구성 요소와 달리 특정 작업 (예: 네트워크 보안 그룹 규칙, 특정 포트를 차단, URL 허용 목록 및 등의 사용자 지정) 못했고 클러스터 (예: 443 아웃 바운드 포트를 차단 하는 방화벽 규칙)를 렌더링할 수 있습니다.

> [!NOTE]
> 송신 (예를 들어 명시적 도메인/포트 whitelisting) 클러스터에서 완전히 잠그는 지원 되는 AKS 시나리오가 아닙니다 지금은 합니다. Url 및 포트의 목록은 경고 없이 변경 될 수 있습니다 하 고 티켓을 통해 Azure 지원에서 제공 될 수 있습니다. 제공 된 목록인 감수 하는 고객에 대해서만 *언제 든 지 클러스터의 가용성을 저하 될 수 있습니다.*

## <a name="alphabeta-kubernetes-features-not-supported"></a>알파/베타 Kubernetes 기능 (지원 되지 않음)

AKS는만 업스트림 Kubernetes 프로젝트 내에서 안정적인 기능을 지원합니다. 그렇지 않으면 전달 하 고 문서화 하지 않는 한 업스트림 Kubernetes에서 사용할 수 있는 알파/베타 기능 지원 되지 않습니다.

두 경우에 알파 또는 베타 기능 수를 롤아웃할 수 GA 이전:

* 고객은 AKS 제품, 지원 또는 엔지니어링 팀을 사용 하 여 충족 하며 이러한 새 기능을 시도 하도록 명시적으로 요청 받은.
* 이러한 기능은 되었습니다 [기능 플래그를 통해 사용 하도록 설정할] [ 2] (명시적 opt in)

## <a name="preview-features--feature-flags"></a>미리 보기 기능/기능 플래그

기능 및 확장된 테스트, 커뮤니티 및 사용자 의견을 필요로 하는 기능에 대 한 새 미리 보기 기능 또는 기능을 기능 플래그 뒤 Microsoft 출시 됩니다. 이러한 기능은 시험판 고려해 야/베타 되며 사용자가 이러한 새 기능을 제공할 수 있도록 노출 합니다.

그러나 이러한 미리 보기/기능 것은 아닙니다 프로덕션용 – Api, 동작 변경, 버그 수정 및 기타 변경 내용 수 하는 기능 플래그는 불안정 한 클러스터 가동 중지 시간에 발생할 수 있습니다. 이러한 기능에 대 한 지원은 버그/문제 보고 하도록 제한 됩니다. 프로덕션 시스템 또는 구독에서 이러한 기능을 사용 하지 마십시오.

> [!NOTE]
> 미리 보기 기능을 사용 하는 Azure에서 적용 **구독** 수준입니다. 일반 작업에 영향을 주는 기본 API 동작을 변경할 수 있습니다 하는 대로 프로덕션 구독에서 미리 보기 기능을 설치 하지 마세요.

## <a name="upstream-bugs-and-issues"></a>업스트림 버그 및 문제

업스트림 Kubernetes 프로젝트의 개발 속도 제공 하지 않습니다 가변 버그 패치할 수 없는 또는 작동 문제를 해결 AKS 시스템 내 고 요구 하는 대신 더 큰 패치를 업스트림 프로젝트 (예: Kubernetes, 작업자 노드/Os 및 커널). 구성 요소 (예: Azure Cloud Provider) 소유 하는 것에 대 한 AKS/Azure 담당자는 커뮤니티에서 업스트림 문제를 해결 합니다.

여기서 기술 지원 문제에는 하나 이상의 업스트림 버그 근본 원인이 파악 하는 경우 AKS 지원 및 엔지니어링에는 다음 항목을 수행 합니다.

* 식별 하 고 이유는 영향을 주는 클러스터 및/또는 워크 로드에 대 한 지원 세부 정보를 사용 하 여 업스트림 버그를 링크 합니다. 고객 문제를 보고 하 고 새 Kubernetes/기타 릴리스는 fix(es)를 포함 하는 경우 참조 필요한 리포지토리/문제에 대 한 링크가 제공 됩니다.
* 잠재적 해결 또는 완화 합니다. -이 경우 문제를 해결할 수 있습니다 경우도 "[알려진 문제](https://github.com/Azure/AKS/issues?q=is%3Aissue+is%3Aopen+label%3Aknown-issue)" 설명 하는 AKS 정리할 수 있도록 합니다.
  * 문제 및 업스트림 버그 링크
  * 해결 방법 및 솔루션의 업그레이드/기타 지 속성에 대 한 세부 정보
  * 업스트림 릴리스 작업 리듬에 따라 포함에 대 한 대략적인 타임 라인
